### Feature engineering — UPI parser

Eight features extracted per applicant from a 6-month statement:

| Feature                        | Formula                                                      |
| ------------------------------ | ------------------------------------------------------------ |
| `avg_monthly_inflow`           | mean of monthly credit totals                                |
| `income_regularity`            | 1 − stdev(salary-credit-day)/15, clipped [0,1]               |
| `savings_ratio`                | (Σ inflow − Σ outflow) / Σ inflow                            |
| `merchant_diversity`           | mean of unique narration prefixes per month                  |
| `discretionary_spend_ratio`    | spend on food/entertainment merchants / total debit          |
| `emi_count`                    | months containing an EMI/loan keyword                        |
| `atm_withdrawal_ratio`         | ATM debit / total debit                                      |
| `p2p_transfer_ratio`           | P2P-style narrations / total debit                           |

These mirror the eight signals validated as "very high" / "high" predictive power in the BharatScore feature spec (`Uco.md` §3).

### Model

- **LightGBM** binary classifier, 15 features (8 UPI + 5 psychometric + age + dependents).
- Training: synthetic data generated by `generate_synthetic_data.py`, where the default label is a logistic function of the same features the model recovers — this is how we know the pipeline is end-to-end correct without real loans.
- **SHAP TreeExplainer** for per-decision explanations (top 6 drivers shown to applicants).
- **DiCE** generates 3 counterfactual paths for Tier-C/D applicants, restricted to *actionable* features (savings, discretionary spend, ATM share, regularity, psychometric scores, EMI count).

### Frontend (`sakhi/frontend`)

- React 18 + Vite, 460-px max-width "phone-frame" layout.
- 6 routes: `/`, `/applicant`, `/upi`, `/test`, `/result`, `/admin/:id?`.
- Two-language i18n via a tiny string table; HTML5 audio with Web Speech API fallback for the psychometric flow.
- Vite dev-proxy forwards `/api/*` to `localhost:8000` so there's no CORS configuration to learn.

## Data flow — happy path

1. User clicks **Start application** → enters name/age/dependents.
2. User picks a UPI statement PDF → frontend POSTs to `/api/parse-upi` → backend returns 8 features in 1–3 s.
3. User answers 10 psychometric items → frontend POSTs full payload to `/api/score`.
4. Backend: psychometric scoring → LightGBM PD → tier assignment → SHAP top drivers → if Tier C/D, DiCE counterfactuals → write `Application` row → return decision JSON.
5. Frontend renders the score, top drivers, counterfactual paths, and behavioral profile.

## What we deliberately did not build (yet)

- Voice assessment, social graph, nudge engine, SHG mode, default insurance pool, MSME track — these are in `Uco.md` and on the Tier 2 / Tier 3 roadmap, not in the MVP scope.
- Real Account Aggregator integration (Setu/Finvu). PDF upload is the demo path; AA is a v2 swap-in for `parse_and_aggregate`.
- DPDP-grade consent ledger. Schema fields exist (`upi_features_json` is per-application) but a separate `consents` collection with grant/revoke + cascade-delete is post-MVP.
- Production TTS. eSpeak NG is functional but rough; production should use Piper TTS with the `hi_IN-pratham` voice.
